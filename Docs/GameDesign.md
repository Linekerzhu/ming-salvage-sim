# 《崇祯模拟器》总设计文档

本文档是《崇祯模拟器》的总策划纲领。它同时回答三个问题：

- 这个游戏模拟什么样的皇帝治理体验。
- NPC、疆域、官僚体系和 LLM 如何组成可推演的游戏机器。
- 第一版应该先做哪条最小可玩的纵切，避免一次性铺开过多系统。

具体 iOS 原生技术栈、模块拆分和量化里程碑见 `Docs/IOSNativeImplementationPlan.md`。

## 设计定位

《崇祯模拟器》是一款以崇祯元年为开局点的历史政治模拟游戏。玩家扮演新即位的崇祯皇帝，在明末财政、边防、党争、灾荒、内廷与地方治理压力中，通过任免官员、召对问策、批阅奏疏、调度财政与处置事件，尝试延缓、改变或重新解释大明的走向。

游戏追求的是“可推演的历史感”，不是逐条复刻史实。史实用于建立可信的制度、人物和环境底座；玩法允许为了可读性、可操作性和节奏而做清晰的抽象。

LLM 时代的策略游戏不应只是把固定按钮换成自然语言输入。这个项目要探索的是一种游戏领域的 LLM Harness：开发者提供可靠的数据基座、规则边界、状态账本和交互界面，LLM 在这些边界里扮演 NPC、主持叙事、审计对话和解释后果。

## 开局时间

默认开局锚点为：

```text
崇祯元年，公元 1628 年
```

所有环境基建、官职设置、行政区划、NPC 初始状态，默认以这一年作为第一版基准。后续如果引入剧本年份，应明确写入独立 scenario 配置，不直接改动 1628 基准数据。

## 核心原则

游戏管理边界：

```text
皇帝能直接管理/对话的官员：止于副部堂级
皇帝能看见/调度的官员：中央到司局级，地方到州府级
县级及以下：作为事件、舆情、税粮、地方叙事和 LLM 语料来源
```

这意味着玩家不会直接管理每个县令、胥吏、士绅、军户或商人，但这些群体会通过奏疏、灾情、案件、地方压力和人物关系进入游戏。

核心治理原则：

```text
皇帝不是全知上帝，而是在战争迷雾中治理国家。
皇帝真正能影响国家机器的方式，是经营少数关键人物。
自然语言奏对必须转化为可追踪的政治目的、承诺、旨意和后果。
LLM 负责表演、解释和审计；核心状态变化必须由本地规则系统裁判。
```

## 核心体验

玩家面对的不是一张完全透明的国家面板，而是一座被奏疏、党争、灾荒、亏空、军情和人情遮蔽的黑箱。

开局的崇祯皇帝对官僚体系并不熟悉。他看得见姓名、官职、传闻和部分履历，却看不见每个人真实的算盘。官员在御前可以忠诚、推诿、试探、迎合、敷衍、暗中反抗，也可以在条件满足时真正替皇帝担责。

游戏的主要乐趣不是在菜单里点出最优解，而是通过召见、追问、许诺、试探、施压、任免、密查和兑现承诺，逐步识人、用人、制衡派系，并把皇帝的意志压进庞大而迟滞的官僚体系。

## 旧原型机制洞察

开发本 iOS 项目前，已经存在一个网页版原型 `Ming`。它不作为素材来源；其中图片、资源、长 prompt 和具体实现都不直接搬入 iOS 项目。但它验证过一批有价值的机制，值得作为设计经验吸收。

旧原型最有价值的不是美术资源，而是这些机制：

- 月度回合：月初奏报、君臣奏对、形成旨意、月末结算、记忆沉淀。
- 奏对目的：玩家一句话不只是聊天，而可能开启一个政治目标。
- 心理握手：官员是否真的承诺，需要经过目的识别、语义审计、心理分和条件判断。
- 承诺账本：官员答应过什么、皇帝答应过什么、条件是否兑现，都成为长期政治事实。
- 执行背书：只有已经达成的承诺，才能在后续旨意中降低阻力；未闭环条件不能算作自愿配合。
- 政治黑板：月末结果要能给出证据链，解释为什么旨意成功、变形、搁置或反噬。
- 动态立绘：角色的身份、官职、服制变化可以触发新的视觉状态，而不是永远使用一张静态头像。
- 严格 LLM 合同：LLM 输出必须经过结构校验；失败时宁可报错或不落档，也不能默默编造状态。

因此，iOS 版应该重建更干净的 Swift 原生机制，而不是照搬 Python/SQLite 架构。

## 第一版核心循环

第一版采用月度回合制。一个月不是简单的“下一回合”，而是一套治理节奏：

```text
月初奏报
  ↓
选择召见对象
  ↓
君臣奏对
  ↓
识别奏对目的与心理握手
  ↓
生成一道旨意草案
  ↓
玩家确认或修改
  ↓
规则系统预评估执行风险
  ↓
月末结算
  ↓
生成月报、记忆卡、承诺账本变化和因果黑板
```

第一版重点不是让玩家一次管理整个帝国所有模块，而是先把“召见一名官员、谈成或谈崩一件事、形成一道旨意、看到后果”这一条链路做扎实。

## 技术与美术基线

- 平台：iOS SwiftUI。
- 当前主界面：群星谱，展示 NPC 档案、天罡、命数、心盘、势网。
- 字体：`LXGWWenKai-Medium.ttf`，统一作为游戏中文字体。
- 文本 LLM：DeepSeek 路线，用于开发期文学/数值支持和游戏内对白、奏疏、事件文本。
- 图像 LLM：Google Gemini 图像路线，用于开发期美术资产和未来游戏内插画、图标、反馈图。
- API Key 不进入仓库；开发期通过本地环境变量或 Xcode Scheme 注入。

## 动态立绘机制

动态立绘属于 v1.5 机制。第一版可以先保留接口思想，不强制实时生成每次变化后的图片。

动态立绘的核心不是“随机画头像”，而是“角色视觉状态跟随政治身份变化”：

- 每个 NPC 应有稳定的 `PortraitDNA`，用于保持五官、年龄气质、身份记忆点和人物辨识度。
- 每张可显示立绘应有 `PortraitAsset`，记录人物、当前身份、官职、服制、生成模型、状态和图片数据。
- 当前身份、官职、品秩、内廷/外朝/后宫/军职/民籍等变化，会形成新的立绘签名。
- 签名变化时，可以排队生成新立绘；生成失败不应阻断游戏结算。
- 升官、罢官、入内廷、脱籍、出仕、受伤、老去等状态变化，都可以在未来成为视觉更新触发点。

Google 图像模型用于生成开发期美术和未来游戏内图像。图像生成必须遵守干净资源策略：不复用旧原型污染素材，不把本机参考图路径或密钥写进仓库。

## 正式行政区划

第一版正式行政区划锚定崇祯元年，做到府/直隶州一级，不展开精确县级行政表。

层级：

```text
大明
  ├─ 北直隶
  ├─ 南直隶
  ├─ 山东
  ├─ 山西
  ├─ 河南
  ├─ 陕西
  ├─ 四川
  ├─ 江西
  ├─ 湖广
  ├─ 浙江
  ├─ 福建
  ├─ 广东
  ├─ 广西
  ├─ 云南
  └─ 贵州
      └─ 府 / 直隶州 / 军民府 / 西南土司性质府州
```

当前数据规模：

```text
根节点：1
一级区：15
府级节点：203
```

府级节点拆分：

```text
普通府：145
直隶州：29
军民府：13
西南土司/羁縻性质府州：16
```

每个府级节点附带 `subordinate_places_text`，作为连续文本地名语料。该字段用于剧情、奏疏、事件落点和 LLM 生成，不作为精确县级行政区划、赋税口径或司法辖区依据。

## 地方常设官职

地方官制采用游戏化简化，不追求完全复刻明制品级。第一版地方常设主官：

```text
从二品：布政使      民政财政
正三品：按察使      司法监察
从三品：都指挥使    军事卫所
正四品：州府主官    知府 / 知州 / 军民府主官 / 土府州主官
```

按照当前行政区划：

```text
15 个一级区 × 3 个省级主官 = 45
203 个府级节点 × 1 个州府主官 = 203
地方常设主官合计 = 248
```

在未来玩法里，布政使、按察使、都指挥使分别承担省级财政民政、司法监察、军事卫所的数值入口；州府主官承担地方执行和事件落点。

## 中央组织架构

中央第一版按玩家可读性简化为：

```text
皇帝
├─ 内阁
├─ 六部
├─ 都察院
├─ 通政司
├─ 大理寺
└─ 内廷
```

六部固定为：

```text
吏部：官员任免、考核、升降、党争人事
户部：财政、税赋、粮草、赈灾、国库
礼部：礼制、科举、外交、祭祀、名分
兵部：军令、边防、调兵、武官任免
刑部：案件、律法、审判
工部：水利、城防、工程、制造
```

六部单部简化编制：

```text
尚书：1
左侍郎：1
右侍郎：1
郎中：4
```

六部合计：

```text
每部 7 人 × 6 = 42 人
```

中央官职原则：

```text
尚书、侍郎等副部堂以上：皇帝可直接召对
郎中等司局级：皇帝可见，可调度、查办、派差，但通常不直接日常对话
```

总督、巡抚属于特别授权或危机系统职位，暂不作为普通常设三司结构的一部分。后续可按剧本、战区、财政或灾荒事件动态设立。

## NPC 数值系统

NPC 是游戏政治推演的核心资产。当前主要数值层：

### 天罡

36 维人物坐标，分政治立场与专业能力。政治维度用于判断价值倾向、站队、政策偏好；专业维度用于判断实际办事能力。

### 心盘

心盘是人物面对皇帝与局势时的动态心理入口。当前初始字段包括：

```text
道合
势合
畏惧
信任系数
仇恨
象限
核心关切
```

未来运行时，心盘应随赏罚、召对、党争、失败、流言和事件后果变化。

### 命数

命数是人物性情和行为模式的中层抽象。当前为 10 轴，用于解释一个人遇压、冒险、守名、信任、结群、记怨等倾向。

### 势网

势网记录 NPC 的关系资本、派系嵌入、恩庇网络、敌对压力和关键关系。它是党争、举荐、弹劾、结盟、背锅和救援的重要基础。

## 奏对目的与心理握手

君臣奏对不是普通聊天框。每轮对话都要先判断玩家是否正在推进一个政治目的。

`AudienceGoal` 表示本次奏对的政治目的，例如：

- 请某官支持一项财政政策。
- 要某官出任某职。
- 让某官密查某案。
- 要某官在廷议中背书。
- 试探某官是否愿意承担风险。

系统需要区分普通问询与真正目的。比如“辽东情势如何”只是问策；“你愿不愿替朕清查辽饷亏空”才可能开启奏对目的。

心理握手状态使用四类：

```text
none         未成约：没有明确承诺，或只是普通聊天。
conditional  附条件：官员提出可审计条件，暂未真正答应。
sealed       握手成功：官员明确承诺，且条件已闭环或无需条件。
blocked      未说服：官员拒绝、回避、心理阻力过高，或关键证据不足。
```

重要规则：

- “臣谨听”“容臣斟酌”“不敢不从”不能自动算握手成功。
- 畏惧只能压制公开抗命，不能变成忠诚或真实同意。
- 高风险事项必须有明确承诺和证据，不能靠 LLM 替官员补事实。
- 附条件不是成功，只是一个等待皇帝兑现条件的政治悬案。

## 承诺账本与履约

握手成功后，系统应把它写入 `AgreementLedgerItem`，也就是承诺账本。

承诺账本记录：

```text
承诺对象
核心议题
承诺标的
承诺类型
风险 Stakes
握手状态
条件状态
标的状态
期限
履约证据
政治效果
```

只有 `target_status=achieved` 的承诺，才能成为后续旨意执行的真实背书。其他状态必须区别处理：

```text
pending_conditions  条件未闭环：不得当作自愿配合。
failed              皇帝失信或条件失败：会伤害信任、派系满意和皇威。
blocked             未说服：若强推，按高压诏令和真实阻力结算。
achieved            标的达成：可降低个人拖延和执行阻力。
```

`AgreementCondition` 是承诺中的可审计条件，例如：

- 拨银多少。
- 给明旨授权。
- 添派人手。
- 保全家眷。
- 给名分、廷议、会审或部议程序。
- 设期限并允许复命。

条件不是文学装饰。它必须能被后续旨意、邸报、月末结算或落库事实验证。

## 条件履约与失信

官员开出条件后，系统要在两个时点复查：

- 旨意发布前后：玩家是否把条件写进旨意。
- 月末结算后：规则结果是否真的兑现条件。

兑现承诺会形成政治资本：

- 提高该 NPC 对皇帝的势合和信任。
- 降低该 NPC 在相关事项上的拖延。
- 小幅提升皇威或派系满意。
- 成为后续同类政策的执行背书。

失信会形成政治债：

- 降低该 NPC 信任。
- 增加仇恨或离心。
- 引发派系借题、清议疑惧、承办人敷衍。
- 在月末因果黑板中提示玩家“这不是随机失败，而是失信导致的政治反噬”。

## 心盘动态变化

心盘不是静态人物标签，而是 NPC 对皇帝的动态心理账。

运行时心盘至少包括：

```text
道合：理念是否接近。
势合：利益和安全是否与皇帝绑定。
畏惧：是否害怕皇权压力。
信任：是否相信皇帝会兑现话。
仇恨：是否记恨皇帝或其执行链。
象限：股肱、权附、道隐、离心等关系状态。
```

奏对、赏罚、强旨、任免、履约、失信、派系波动都会改变心盘。

关键原则：

- 道合决定理念靠近，势合决定利益靠近。
- 畏惧不是忠诚，只能压住公开抗命。
- 高畏惧、低信任、强仇恨的人，可能表面服从、暗中拖延。
- 皇帝兑现条件会增加信任；皇帝强推或失信会制造长期后果。

## 环境数值系统待建

行政区划和官僚架构只是环境基建第一层。后续需要补运行时环境变量：

- 国库
- 粮储
- 税赋压力
- 民心
- 灾荒
- 流民
- 治安
- 边防压力
- 官场腐败
- 朝堂派系温度
- 内廷干政强度
- 奏疏拥堵

这些变量应挂在不同层级：

```text
全国级：国库、总体民心、朝堂温度、边防总压
一级区：省级财政、灾荒、治安、赋税、兵备
府州级：地方事件、民变风险、粮价、案件、官声
```

## 游戏运行规则基座

游戏运行规则基座负责把自然语言和叙事结果落到真实状态上。它至少包括：

- 回合推进规则：月份、年号、历史锚点、月初月末流程。
- 旨意规则：玩家自然语言转成 `ImperialDirectiveDraft`，确认后成为可结算旨意。
- 财政规则：国库、内库、税赋、拨款、亏空、赈灾、军饷。
- 官僚执行规则：官署完整度、承办人能力、官职适配、派系阻力。
- 心盘规则：奏对和履约如何改变 NPC 对皇帝的心理状态。
- 记忆规则：重要对话、旨意、承诺、失败和背叛如何沉淀。
- 审计规则：LLM 输出如何通过结构校验后才进入游戏事实。

## 执行预评估

一道旨意发布前，系统应生成 `ExecutionAssessment`，用来预判这道旨意的执行风险。

执行预评估综合：

- 相关官署班子是否完整。
- 承办人是否在任、是否适合该领域。
- 承办人天罡能力与专业倾向。
- 承办人心盘象限、势合、畏惧、信任和仇恨。
- 该 NPC 所属派系满意度和影响力。
- 本月奏对立场：支持、附条件、反对或未定。
- 承诺账本里是否存在已达成的执行背书。
- 国库、粮储、地方压力等硬盘面是否支持该旨意。

预评估不是最终结果，而是给月末结算和玩家判断提供证据。玩家可以在发布前看到风险提示，也可以选择强推。

## 政治因果黑板

月末结算后，玩家需要知道“为什么如此”。

`CausalityNote` 负责把已经发生的事实整理为可读因果，而不是重新模拟一次。它应包含：

- 哪些官员支持、附条件、反对或未定。
- 哪些承诺已经达成，哪些仍未闭环。
- 国库、粮储、派系、官署空缺、地方压力造成了什么折损。
- 哪些旨意成功推进，哪些变形、拖延、失败或反噬。
- 哪些结果会成为后续记忆、传闻、党争或官员心理变化。

政治因果黑板是玩家学习游戏规则的关键。它让失败不显得随机，也让成功不只是 LLM 写得好听。

## 记忆系统

记忆系统把一次次奏对和结算变成长期历史。

记忆卡应记录：

```text
主体：人物、机构、地区、议题或国家。
事件类型：奏对、旨意、任免、承诺、失信、密查、灾情、财政。
原因、过程、结果。
情绪倾向。
重要度。
标签。
来源证据。
有效期或永久保留。
```

普通记忆会随时间衰减，重大政治事件永久保留。LLM 在生成对白、奏疏和月报时，只能使用当前可见且相关的记忆，避免把旧事无限放大。

## LLM Harness 与使用原则

LLM 不直接决定核心数值真相。它主要用于：

- 生成奏疏、对白、事件叙事、传记和命批
- 辅助提出数值方案
- 为美术图像生成 prompt
- 作为数据校验的提醒源

在运行时，LLM 的职责进一步细分：

```text
NPC 扮演：根据天罡、心盘、官职、派系和记忆生成官员回复。
奏对审计：判断玩家目的、NPC 是否承诺、条件是否明确。
旨意润色：把玩家意图整理成符合时代语感的旨意草案。
月报叙事：把规则结算结果写成邸报、奏报和旁白。
图像提示：根据人物 DNA、身份和服制生成图像 prompt。
```

Swift 本地规则系统负责：

```text
真实数值变化
状态合法性校验
承诺账本落档
条件履约复查
心盘变化
存档
失败处理
测试与回归
```

LLM 输出必须进入人工、脚本或结构化合同校验流程。历史事实争议不能只听一个模型；DeepSeek 可以提醒，必要时再用 Google/Web 或其它资料交叉复核。

## 第一版纵切

第一版可玩纵切锁定：

```text
财政危机
君臣奏对
一道旨意
月末结算
承诺账本
政治因果黑板
```

第一条示例玩法：

```text
月初：户部奏国库亏空、辽饷吃紧、陕西需赈。
玩家：召见户部或内阁关键官员。
奏对：询问亏空、试探是否愿意清丈或筹饷。
握手：官员可能支持、附条件、推诿或反对。
旨意：玩家确认一条财政旨意，例如清查亏空、拨赈、加派、追赃。
结算：系统根据财政硬盘面、承办人、官署、派系和承诺账本裁判结果。
月报：给出国库变化、民心变化、官员心盘变化和因果说明。
```

暂不在第一版同时铺开完整军事、外交、后宫、地图和实时动态立绘生产化。它们可以进入后续阶段，但不能稀释第一条核心链路。

## 当前已确认事项

- 游戏开局时间：崇祯元年，1628 年。
- 正式行政区划做到府/直隶州一级。
- 州府同级，州府主官统一正四品。
- 地方一级区使用北直隶、南直隶、十三布政司。
- 地方常设主官按 15 个一级区和 203 个府级节点计算，共 248 人。
- 皇帝直接管理止于副部堂级。
- 皇帝可见范围：中央到司局级，地方到州府级。
- 县级及以下作为语料、事件和环境压力来源，不作为常设可管理官位。
- 第一版核心场景优先做君臣奏对，朝堂会议后置。
- 第一版治理模块优先做财政危机纵切。
- LLM 不直接改核心数值，必须经过规则裁判和结构校验。
- 旧网页版 `Ming` 只作为机制原型参考，不作为资源来源。
- 动态立绘作为 v1.5 机制写入总纲，v1 先保留接口思想。
